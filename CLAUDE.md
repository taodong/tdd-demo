# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Goal
Create a website to manage user authorizations and privileges.
## Technology Stack
Please prioritize the following technologies in coding
- Java 25
- Spring Boot 4.0.6
- Flyway
- H2 Database Engine
- Lombok
- Maven
- Node
- Vue 3
- Bootstrap 5.3.8
- TypeScript
## Architecture Principle
- Front end should be a standalone node implementation which can be run independently through node
- All communications between front end pages and back end business logic are through RESTful APIs
- Backend uses a thin-layer structure with controller, service, and repository structure if possible
- For backend, try to use Lombok annotations if possible
- Use `yaml` format for any configurations if possible
- The project should be deployed through Maven
## Repository Layout

This repo contains specifications and test infrastructure only — no application source code yet. The application must be built here following the architecture principles above.

| Directory | Purpose |
|-----------|---------|
| `features/` | Feature specification docs describing UI, behavior, and tasks to implement |
| `tests/` | Acceptance test definitions (Markdown) executed by the Waterwheel QA agent |
| `outputs/` | Generated by test runs: `test-results.json`, `test-context.json`, per-test `*_log.json` |
| `waterwheel-instructions/` | Config for the QA container: global context vars, allowed domains, system prompt |

Tests in `tests/` may declare `required: <id>` to express ordering dependencies (e.g., `2-lock-user-account.md` requires test 1 to pass first). The global test variable `TEST_URL` resolves to `http://host.docker.internal:8080`, which maps to localhost:8080 from inside the Docker container.

## Application Commands

Once the application source code exists under this directory:

```bash
# Backend
mvn clean install          # Full build + unit tests
mvn spring-boot:run        # Start backend on port 8080

# Frontend (from frontend directory)
npm install
npm run dev                # Dev server
npm run build              # Production build
```

Flyway migrations live in `src/main/resources/db/migration/`. H2 runs in-memory by default; configure via `src/main/resources/application.yml`.

## Development Workflow

1. Read the relevant feature doc in `features/` to understand requirements and tasks.
2. Implement backend (controller → service → repository) and frontend (Vue 3 components).
3. Ensure `mvn clean install` passes.
4. Confirm QA environment, then run acceptance tests.

## Skills
### Confirm QA Environment
The QA environment is configured through a Docker container. The QA environment is ready when all three conditions below are met:
1. The container is running: `docker ps --filter "name=waterwheel-test-agent" --format "{{.Names}} {{.Status}}"`.
2. The `tests` folder exists and there is at least one markdown file inside it.
3. The `outputs` folder exists.
### Feature Acceptance Test
When performing a feature acceptance test or prompted to verify the feature, use the following steps:
1. Start the Spring Boot application locally on port 8080.
2. Run command `docker container exec waterwheel-test-agent run-qa > qa.log 2>&1 &` and capture its PID with `echo $!`.
3. Every 5 minutes, check whether the process is still running (`kill -0 <PID>`).
4. When the process exits, stop the web application.
5. Read `outputs/test-results.json`. The feature is accepted only if all the tests listed completed with success status.
**Do not run `docker stop`, `docker kill`, `docker rm`, or `docker restart`. Only exec into the container.**
### Feature Debug
When an acceptance test fails, identify the bug in the code through the process below:
1. Find the failed test reported by `outputs/test-results.json` and extract the `file` name.
2. Find the test context file `outputs/test-context.json` if available.
3. Locate the test instruction file under `tests` matching the `file` property.
4. Locate the individual test log file under `outputs`. The log file name is the `file` name without extension plus `_log.json`. For example, if the `file` in step 1 has value `test-wikipedia-english.md`, the test log file is `test-wikipedia-english_log.json`.
5. Investigate the content of the test instruction, test log, `test-results.json`, and `test-context.json` to identify the defect in code which caused the test failure.
6. If a bug is identified, report it then implement the fix. Otherwise, report preliminary findings and wait for further instructions.
